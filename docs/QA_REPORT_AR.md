# تقرير اختبار شامل لموقع الأمين للبرجولات

تاريخ التنفيذ: 2026-03-13

## 1) نطاق الاختبار
- مراجعة بنية الواجهة الأمامية (React + Vite) وتحليل عناصر التفاعل والروابط.
- فحص أولي للأداء وقابلية الاستخدام وإمكانية الوصول (Accessibility) من خلال مراجعة الكود.
- مراجعة أولية لجوانب الأمان على مستوى الواجهة الأمامية.
- فحص جميع الروابط والوظائف الظاهرة في الكود.

> ملاحظة مهمة: لم يكن بالإمكان تشغيل المشروع محليًا بسبب منع تنزيل حزم npm (خطأ 403 من registry)، لذلك تعذر تنفيذ اختبارات تشغيلية حيّة (E2E) وعبر متصفحات/أجهزة فعلية.

## 2) الأوامر المنفذة ونتائجها
1. `npm ci`  
   - **النتيجة:** فشل (لا يمكن استخدام `npm ci` مع lockfile الحالي).
2. `npm install`  
   - **النتيجة:** فشل (403 Forbidden عند تنزيل `@google/genai`).
3. `npm run lint`  
   - **النتيجة:** فشل نتيجة غياب الاعتماديات (react/motion/lucide وغيرها غير مثبتة).
4. `rg -n "href=|id=\"|window.open|onKeyPress|GoogleGenAI|target=\"_blank\"|mailto:|tel:" src/App.tsx`
   - **النتيجة:** نجح، وتم استخراج الروابط والوظائف للتحقق اليدوي.

## 3) نتائج اختبار الوظائف والروابط (Static Functional Review)
### 3.1 الروابط الداخلية (Anchor Navigation)
- الروابط الموجودة: `#home`, `#services`, `#gallery`, `#about`.
- الأقسام المقابلة موجودة فعليًا في الصفحة (`id` مطابق).
- قسم `#contact` موجود، لكن لا يظهر ضمن روابط التنقل العلوية المستخرجة؛ قد يكون هذا مقصودًا تصميميًا.

**الحالة:** مقبول مبدئيًا (تحقق بنيوي فقط).

### 3.2 الروابط الخارجية ووسائل الاتصال
- WhatsApp: مستخدم في عدة مواضع عبر `https://wa.me/${PHONE_NUMBER}`.
- Facebook وInstagram موجودان مع `target="_blank"` و `rel="noopener noreferrer"` (ممتاز أمنيًا).
- روابط الهاتف `tel:` موجودة في أكثر من موضع.
- إرسال نموذج التواصل يفتح WhatsApp عبر `window.open` مع رسالة مكوّنة من بيانات النموذج.

**الحالة:** جيدة بنيويًا، وتحتاج تحققًا تشغيليًا حيًا بعد توفر التثبيت.

## 4) تقييم الأداء (Performance)
### ملاحظات
1. الصور كثيرة في المعرض، بدون `loading="lazy"` للصور أسفل الطية (below-the-fold)، مما قد يرفع زمن التحميل الأولي.
2. لا يظهر من الكود وجود تحسينات صور (WebP/AVIF أو أحجام responsive عبر `srcset`).
3. وجود تأثيرات حركة متعددة (`motion`) قد يؤثر على الأجهزة الضعيفة إذا لم تُحكم بعناية.

### توصيات
- إضافة `loading="lazy"` و`decoding="async"` لصور المعرض.
- توفير نسخ صور مضغوطة (WebP/AVIF) مع `srcset`.
- تفعيل/تخفيف الأنيميشن بناءً على `prefers-reduced-motion`.

## 5) تقييم قابلية الاستخدام (Usability + UX)
### ملاحظات
1. نموذج التواصل لا يحتوي تحققًا واضحًا لحقول الإدخال (مثل إلزام الاسم/الرقم وصيغة الرقم).
2. أزرار الأيقونات الاجتماعية لا تحتوي نصًا بديلًا واضحًا للقارئات (`aria-label`).
3. واجهة RTL عربية مناسبة بشكل عام، لكن يلزم اختبار حي للتباين اللوني والتركيز عبر لوحة المفاتيح.

### توصيات
- إضافة Validation صريح للحقول + رسائل خطأ مفهومة.
- إضافة `aria-label` لأزرار/روابط الأيقونات فقط.
- اختبار tab order وfocus states بعد التشغيل.

## 6) تقييم الأمان (Frontend Security Review)
### ملاحظات/مخاطر
1. استخدام Gemini API مباشرة في المتصفح عبر `GoogleGenAI` مع `process.env.GEMINI_API_KEY` يخلق خطر كشف المفتاح (أو فشل البيئة لأن Vite يتوقع `import.meta.env`).
2. لا توجد مؤشرات في المشروع الحالي على طبقة Backend Proxy لاستدعاء الذكاء الاصطناعي بشكل آمن.
3. استخدام `window.open` جيد نسبيًا هنا لأنه موجّه لـ `wa.me`، لكن من الأفضل إضافة ضوابط موحدة للروابط الخارجية.

### توصيات إصلاح
- نقل استدعاء Gemini إلى Backend endpoint (Express مثلاً) وحفظ المفتاح في بيئة الخادم فقط.
- في الواجهة، استدعِ endpoint داخلي بدل استدعاء Gemini مباشرة.
- اعتماد سياسة رؤوس أمان في النشر (CSP, X-Frame-Options, Referrer-Policy, Permissions-Policy).

## 7) التوافق مع المتصفحات والأجهزة
### الوضع الحالي
- **غير مكتمل تشغيليًا** بسبب تعذر تثبيت الاعتماديات وتشغيل التطبيق.

### خطة تنفيذ عند فك الحظر
1. تشغيل اختبارات E2E على:
   - Chromium, Firefox, WebKit.
2. اختبار أحجام الأجهزة:
   - Mobile (360x800), Tablet (768x1024), Desktop (1366x768+).
3. سيناريوهات حرجة:
   - التنقل بين الأقسام.
   - إرسال نموذج التواصل وفتح WhatsApp.
   - فتح روابط Facebook/Instagram/Phone.
   - فتح/إغلاق قائمة الجوال والـ Chatbot وإرسال رسالة.
4. تشغيل Lighthouse (Performance/Accessibility/Best Practices/SEO).

## 8) قائمة المشكلات المحتملة وأولوية الإصلاح
1. **حرج (Critical):** مفتاح Gemini في الواجهة الأمامية (أمان).
2. **عالٍ (High):** غياب تحقق النموذج (جودة بيانات + UX).
3. **متوسط (Medium):** غياب lazy loading وتحسين الصور (أداء).
4. **متوسط (Medium):** تحسينات وصول (`aria-label`, keyboard focus) (Accessibility).
5. **منخفض (Low):** `onKeyPress` قديم نسبيًا؛ يفضل `onKeyDown`.

## 9) ملخص تنفيذي
- بنية الروابط والأقسام الأساسية تبدو متماسكة من خلال مراجعة الكود.
- توجد ملاحظات مهمة في الأمان والأداء وإمكانية الوصول.
- التقرير الحالي **Static + Audit-based** وليس تشغيلًا حيًا بسبب قيود بيئة الاعتماديات.
- بعد إتاحة تثبيت الحزم، يلزم تنفيذ جولة E2E + Cross-browser + Lighthouse لإغلاق التقييم النهائي.
